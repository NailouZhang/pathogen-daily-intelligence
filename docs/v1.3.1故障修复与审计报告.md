# v1.3.1 故障修复与审计报告

## 1. 故障现象

真实检索在多模型翻译兜底结束后失败：

```text
AttributeError: 'NoneType' object has no attribute 'update'
```

触发位置为 `pipeline.py::_translate_remaining_items()` 对 `translation_audit` 执行 `.update()`。

## 2. 根因

规范化对象在 v1.3 中使用：

```python
"translation_audit": None
```

兜底代码使用：

```python
item.setdefault("translation_audit", {}).update(...)
```

`dict.setdefault()` 只在键不存在时写入默认值；键已经存在但值为 `None` 时，它会返回 `None`。因此当 GitHub Models、Gemini 和 Groq 均未生成通过校验的翻译，记录进入最终兜底阶段时发生崩溃。

## 3. 影响条件

需要同时满足：

1. 运行启用了 LLM；
2. 至少一条记录缺少已验证中文翻译；
3. 所有可用模型均失败、缺少权限、限流或输出未通过校验；
4. 记录的 `translation_audit` 为 `null`。

因此 Demo 或已有翻译记录可能不触发，而真实检索更容易触发。

## 4. 修复内容

### 4.1 新对象不再产生空审计值

`ScholarlyWork` 和 `NewsArticle` 新建时使用：

```python
"translation_audit": {}
```

### 4.2 兼容旧历史数据

新增 `ensure_dict_field()`：

- 字段已是对象时原样复用；
- 字段为 `null`、列表、字符串、数字或其他异常类型时，安全替换为空对象；
- 可选写入默认值；
- 不修改来源事实字段。

### 4.3 加固全部可变审计字段

统一处理：

- `translation_audit`
- `processing_audit`
- `retrieval_audit`
- `display_summary`
- `content`
- `abstract`
- `quality`
- `fingerprints`

### 4.4 最终编排级兜底

即使未来 LLM 编排层出现未预期的 Python 异常，也不会阻止日报发布。系统将：

1. 记录异常类型和消息；
2. 保留已有中文翻译；
3. 对无翻译记录生成诚实的中英文占位状态；
4. 继续输出 JSON、HTML、邮件、RSS、artifact 和 `intelligence-data`；
5. 在 `llm_runs.jsonl` 中写入 `orchestration_failed_fallback_published`。

这不是静默忽略错误：Workflow 可继续发布，但页面和审计文件会明确说明 AI 层异常。

## 5. 回归测试

新增测试覆盖：

- `translation_audit: null`；
- `processing_audit: null`；
- 异常类型审计字段；
- GitHub Models、Gemini、Groq 全部失败；
- 模型凭据全部缺失；
- LLM 编排函数主动抛出异常；
- 新规范化对象审计字段可直接修改；
- 输出仍然包含完整日报和静态页面。

CI 新增“无模型凭据但 LLM 路径开启”的命令级烟雾测试。

## 6. 提示词和模型路由审查

本次故障不由提示词内容引起，生产提示词和以下路由保持不变：

```text
翻译/单篇分析：GitHub Models → Gemini → Groq → deterministic
每日综合：Gemini → GitHub Models → Groq → deterministic
```

`prompts/` 和 `prompt_review/production_prompts/` 仍由 CI 检查逐字节一致。

## 7. 剩余动态验证

本地环境无法代表你的真实 GitHub Models、Gemini 和 Groq 账户权限。推送后仍需运行一次真实 Workflow，重点检查：

- 模型目录发现；
- GitHub Models 权限和额度；
- Gemini/Groq Secret；
- 新闻网页可访问性；
- API 限流；
- 审计文件中的尝试链。

即使这些动态服务全部不可用，本版也应继续发布无 AI 日报。
