# v1.3 全流程：收集、处理、理解、总结与审计

本文档描述系统设计和已实现的运行路径，不代表每个外部来源在每次运行中都必然成功。动态接口、模型和网页可访问性以每次 Workflow 的审计结果为准。

## 1. 总体原则

系统把任务分成四类责任：

1. **来源接口与网页提供事实材料**：文献元数据、摘要、开放全文片段、官方通报或新闻正文。
2. **Python 负责确定性处理**：查询计划、抓取、规范化、去重、事件聚类、规则过滤、缓存、校验、渲染和审计。
3. **大模型负责受约束的理解**：翻译、结构化抽取、证据化摘要与跨条目综合。
4. **验证层决定模型输出是否可用**：模型不是事实来源；输出不通过校验就进入下一提供商，全部失败后保留原文和规则结果。

同一任务不会并行调用多个模型投票。模型按配置顺序串行尝试，前一个结果通过后立即停止。

## 2. 每日流水线

```text
PathogenProfile 与已批准中英文词典
    ↓
查询组与数据源专用查询计划
    ↓
PubMed / Europe PMC / Crossref / Semantic Scholar
官方来源 / ReliefWeb / GDELT / Google News RSS
    ↓
原始来源记录与请求审计
    ↓
规范化 ScholarlyWork 与 NewsArticle
    ↓
文献实体去重 / 新闻文章去重
    ↓
公开正文与开放全文证据增强
    ↓
规则实体识别、相关性和版面价值分级
    ↓
NewsArticle → PublicHealthEvent 聚类及跨日更新
    ↓
模型结构化理解与翻译（逐任务顺序兜底）
    ↓
Evidence ID、数字、实体、结构与支持项校验
    ↓
DailyIssue 综合
    ↓
GitHub Pages / Streamlit / JSON / RSS / 邮件
    ↓
intelligence-data 与 recovery artifact 审计归档
```

## 3. 文献收集与处理

### 3.1 多源发现与补全

- **PubMed**：生物医学主发现源，获取 PMID、摘要、MeSH、作者和出版元数据。
- **Europe PMC**：独立发现、PMCID、开放获取状态、预印本和摘要补全。
- **Crossref**：DOI、出版社、卷期页、许可和更新关系补全，并执行高精度补充发现。
- **Semantic Scholar**：跨库标识符、摘要和引用关系补充。

四个来源先分别保存为来源记录，再合并为一个 `ScholarlyWork`。合并按标识符、书目字段和关系类型进行；预印本、正式论文、更正与撤稿不会被粗暴删除。

### 3.2 摘要与开放全文证据

默认使用标题和摘要。若记录含 PMCID，系统可从 Europe PMC Open Access `fullTextXML` 获取可公开处理的 XML，并只保留有限长度的以下证据片段：

- Methods / Materials；
- Results / Findings；
- Discussion；
- Conclusion；
- Limitation。

开放全文片段获得 `F1...Fn` 证据编号，摘要获得 `A1...An`。系统不把整篇出版物正文复制到日报仓库。

### 3.3 文献理解

只有进入 `headline`、`brief` 或 `review` 且具有证据句的文献进入深度分析。模型必须输出：

- 中文标题与忠实摘要翻译；
- 研究问题、研究类型、设计、样本或数据；
- 方法；
- 关键发现和定量结果；
- 研究意义及适用范围；
- 作者明确报告的局限；
- 当前证据范围造成的信息缺口；
- 证据强度和证据覆盖范围。

每个实质性结论必须绑定 A/F/T 证据 ID。只有标题而没有摘要或全文时，不允许生成研究结果。

## 4. 新闻与官方通报收集、正文抓取和理解

### 4.1 发现层与事实层分离

GDELT、Google News RSS 和其他聚合器用于发现。系统尽量解析到原始发布 URL，并保留发现来源和最终 URL。官方源和机构原文具有更高优先级，但媒体报道不会自动被当成官方事实。

### 4.2 正文抓取

对优先记录执行有限并发的公开页面抓取：

1. 跟随正常 HTTP 重定向；
2. 优先读取 JSON-LD `articleBody`；
3. 再读取 `article`、`main`、`articleBody` 等语义容器；
4. 最后使用正文段落回退；
5. 拒绝非 HTML、空页面、明显登录墙或订阅拦截页；
6. 不绕过访问控制、付费墙、robots 限制或登录；
7. 仅保存用于分析的有限证据句，不保存完整出版页面。

正文证据获得 `N1...Nn`。抓取成功、失败、解析方法、最终 URL、字符数和截断状态都写入 `retrieval_audit.content_fetch`。

### 4.3 官方通报理解

官方通报提示词区分：

- 事件日与报告日；
- 确诊、可能、疑似病例和死亡；
- 实验室发现；
- 官方行动；
- 来源自身的风险评估；
- 公众指导；
- 相对历史事件的新增变化。

风险评估必须归因于来源，不能写成系统自行作出的风险判断。

### 4.4 媒体新闻理解

媒体提示词必须区分：

- 记者或媒体的陈述；
- 引用的权威机构；
- 已找到的官方确认；
- 尚待核实的主张；
- 转载或引用链；
- 耸动表达。

媒体使用“暴发”一词，不等于官方已经宣布暴发。

## 5. 模型路由与兜底

### 5.1 翻译与单条分析

默认顺序：

```text
GitHub Models
    ↓ 仅在请求失败、JSON 无效、条目缺失或校验失败时
Gemini
    ↓ 仍失败时
Groq
    ↓ 全部失败时
deterministic：保留原文和规则结果
```

GitHub Models 用较大的批次完成快速首轮翻译；Gemini 使用较小批次修复未完成或校验失败记录；Groq 逐条做最后修复。已经成功的记录不会重复发送给后续模型。

### 5.2 每日综合

每日综合偏向质量，默认：

```text
Gemini → GitHub Models → Groq → deterministic
```

综合模型只接收已经去重、聚类、过滤并校验的结构化条目，不直接接收几十个原始网页。

### 5.3 缓存

翻译和结构化分析的缓存键包含：

- 任务；
- 对象 ID；
- 原始标题；
- 摘要/正文证据；
-提示词输入哈希。

相同输入在最近 7 天重叠检索中直接复用。原文发生变化时自动生成新缓存键。

## 6. 模型输出校验

### 6.1 结构校验

- 必须是 JSON 对象；
- 必填字段、类型和枚举受提示词和对象 Schema 约束；
- 批量翻译必须返回每个 `record_id`。

### 6.2 证据校验

- 每个 Evidence ID 必须存在于实际输入；
- 研究结论、病例数、官方行动、实验室发现等必须带证据 ID；
- 每日综合的 `supporting_item_ids` 必须引用真实事件或文献 ID。

### 6.3 数字与科学符号校验

- 原文数字、百分数、单位和比较符号不得在忠实翻译中消失；
- 输出定量结果必须能在所引证据中找到；
- `G<sub>N</sub>` 等科学标签在发送前转换为受保护占位符，输出后再恢复；
- 占位符丢失或改变时，本次输出判定失败。

### 6.4 实体校验

病原、宿主、地点和机构必须：

- 出现在输入证据；或
- 存在于已批准词典/来源注册表。

模型不能自行创造正式病原名称。

### 6.5 失败处理

```text
当前模型输出
    ↓
结构 / Evidence / 数字 / 实体 / 翻译完整度校验
    ├── 通过：保存并停止
    └── 失败：记录原因，调用下一提供商
                    ↓
              全部失败：中文标记不可用，保留英文原文
```

## 7. 事件聚类与跨日更新

新闻文章先去重，再按病原、地点、时间和事件类型生成候选事件。人类病例与动物/宿主监测有硬边界。相同事件跨日保持稳定 `event_id`，并记录：

- 病例状态变化；
- 新增病例或死亡；
- 新实验室结果；
- 地点修正；
- 官方否认或数字更正；
- 新公共卫生行动。

没有实质变化的转载不再次进入主版面。

## 8. DailyIssue 综合

日报综合仅使用通过验证的 `ScholarlyWork`、`PublicHealthEvent` 和结构化分析，生成：

- 今日概览；
- 主头条候选；
- 重大事件；
- 研究亮点；
- 地理、病原和宿主信号；
- 矛盾与待观察项；
- 来源健康；
- 数据质量说明。

趋势语言必须有历史统计支持，否则只能写“单日信号”或“值得观察”。

## 9. 审计输出

每次 Workflow 额外生成：

```text
data/audit/content_enrichment.json
    新闻正文与 Europe PMC 开放全文抓取结果

data/audit/llm_runs.jsonl
    每次模型请求、模型、状态、重试、验证错误和回退链

data/audit/object_audit.jsonl
    每个文献/新闻对象的翻译、分析和正文抓取审计

output_manifest.json
    本次所有输出路径
```

对象中的 `translation_audit.attempt_chain` 和 `processing_audit` 可用于回答：

- 首轮用了哪个模型；
- 为什么没有接受；
- 后续用了哪个模型；
- 最终哪一模型的结果通过；
- 是否使用缓存；
- 新闻是否抓到正文；
- 文献是否使用开放全文；
- 哪些结论因校验失败而未发布。

## 10. 人工审查重点

生产运行后优先抽查：

1. `review` 决策的事件和文献；
2. 病例数字冲突；
3. 同一事件的错误合并或拆分；
4. `title_or_snippet_only` 的新闻分析；
5. 经过第二或第三模型兜底的翻译；
6. `unsupported_claim_count > 0` 的模型输出；
7. 新病原名、新宿主或新地区候选；
8. 每日综合中的趋势措辞；
9. 来源抓取失败或正文极短记录；
10. 更正、撤稿和预印本转正式发表关系。


## v1.4 日期、正文和动态模型补充

- 文献是否进入当期以 online/electronic、数据库 created/indexed/entry/deposited 等“当前可获得日期”为主；未来 print/issue 日期只保存为书目信息，不用于判定当期新增。
- 新闻先进行多策略正文提取，再聚焦保留与目标病原直接相关的证据句。正文、RSS 摘要和标题三种覆盖级别必须明确区分。
- 背景提及不会创建公共卫生事件；病例数字与地点只从明确包含目标病原或疾病的证据句抽取。
- Groq、GitHub Models 和 Gemini 均在运行时发现可用模型。一个模型返回无效结构或未通过证据校验后，会尝试同一提供商的下一个模型，再进入下一提供商。
