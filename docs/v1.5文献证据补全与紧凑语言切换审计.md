# v1.5 文献证据补全与紧凑语言切换审计

## 1. 目标

本版将“发现到一篇文献”与“已获得足够证据进行解读”分开。摘要或全文暂缺时，文献仍然入库、去重、展示和重试；只是禁止生成研究结论。

## 2. 证据等级

- **E0 metadata_only**：只有题名与书目元数据。允许翻译题名，不允许生成摘要、方法、发现、定量结论或意义。
- **E1 abstract**：存在数据库摘要。允许摘要级解读，但不能声称已审阅完整方法、结果、讨论或作者局限。
- **E2 partial_full_text**：存在经文献身份和文本质量校验的有限 HTML、PDF 或纯文本证据。必须说明正文不完整。
- **E3 structured_full_text**：存在 PMC/Europe PMC JATS XML、NCBI PMC BioC 或同等结构化正文证据。
- **E4 full_text_plus_supplement**：为未来扩展保留；只有补充材料也被合法获取和核验时才可使用。

## 3. 内容补全链

1. PubMed、Europe PMC、Crossref、Semantic Scholar 精确标识符查询并补摘要。
2. Europe PMC `fullTextXML`。
3. NCBI PMC BioC JSON。
4. Crossref `link` 中的 XML、HTML、PDF 或文本挖掘链接。
5. Unpaywall 开放位置。
6. Semantic Scholar `openAccessPdf`。
7. DOI 落地页中的 `citation_abstract`、JSON-LD、正文容器、`citation_pdf_url` 或全文链接。
8. PDF：PyMuPDF，失败后 pypdf；OCR 默认关闭，仅在系统已有 Tesseract 且配置启用时限页执行。
9. 全部失败：保留 E0 并进入第 1、3、7、14 天补抓队列。

## 4. PDF 安全与身份校验

- 不绕过登录、付费墙、验证码或访问控制。
- 仅处理直接公开可访问或明确提供给文本挖掘/开放访问的 URL。
- 校验 DOI、题名相似度与作者姓氏；不匹配的 PDF 拒绝使用。
- 检查文件头、大小、页数、文本长度和可打印字符比例。
- PDF 仅在 GitHub Runner 临时内存/临时目录中解析，不写入 `intelligence-data`；只保留 SHA256、来源、许可/版本、有限证据和审计。

## 5. 重试队列

状态存放在：

```text
data/state/seen_items.json → scholarly_recovery_queue
```

队列只保存书目元数据和已有的有限摘要，不保存 PDF 或完整受版权保护正文。E0 和 E1 可分别补摘要或补更强正文证据；达到重试次数或超出最大年龄后停止自动补抓，但文献记录仍保留。

## 6. 大模型边界

模型输入包含 `content_retrieval_status`、`evidence_level`、`acquisition_status`、`extraction_method`、`content_version` 和证据句 ID。

- E0：只可翻译 T0；其他分析字段必须为空。
- E1：只能使用 T0/A*。
- E2：只能使用提供的 F*，并声明正文不完整。
- E3/E4：仍只能使用已提供的证据句，不得假装审阅未传入的全文或补充材料。

## 7. 卡片语言切换

GitHub Pages 的每个双语卡片采用右上角小字符：

```text
en  → 点击显示英文，字符变为 zh
zh  → 点击恢复中文，字符变为 en
```

字符采用绝对定位，不占正文行；打印时隐藏。Streamlit 使用右侧窄列中的 `zh/en` 分段控件。

## 8. 审计字段

每篇文献保留：

- `evidence_acquisition.status`
- `evidence_acquisition.evidence_level`
- `attempt_count`
- `reason_codes`
- `failed_stages`
- `attempts`
- `full_text.source`
- `full_text.content_version`
- `full_text.license`
- `full_text.extraction_method`
- `full_text.sha256`
- `full_text.temporary_file_persisted=false`

这使页面内容、模型分析和原始获取证据之间可以回溯。
