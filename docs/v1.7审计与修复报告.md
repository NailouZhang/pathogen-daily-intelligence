# v1.7 审计与修复报告

本轮修复基于对 2026-07-15 实际生成的《汉坦病毒每日情报》页面的逐条人工审计，
聚焦用户反馈的六个环节：文献获取、文献解析、文献去重、模型解读、解读翻译、网页布局。

## 1. 网页布局：今日要闻与分类板块 100% 重复

**现象**：当天全部 4 篇入选文献同时出现在“今日要闻”和“学术文献”两个板块，内容逐字重复。

**根因**：`issue.py::build_daily_issue` 中，“今日要闻”按分数选出 top N 条目 id，
随后“疫情与公共卫生”“学术文献”两个板块又各自独立地把 headline+brief 全量条目
再放进去一次，从未从中排除已经进入“今日要闻”的 id。当天入选文献数量较少、
普遍得分较高时，几乎全部内容都会被“今日要闻”选中，导致下方板块变成纯复制。

**修复**：分类板块的 `item_ids` 现在会显式排除已经出现在 `lead_item_ids` 中的条目
（`src/pdi/issue.py`）。

## 2. 模型解读 × 翻译：审计链自相矛盾，且拖累翻译成功率

**现象**：审计栏显示 `翻译：github_models · passed · github_models:passed →
github_models:failed → gemini:failed → groq:failed`——同一条目、同一 provider
先“passed”又“failed”，同时该条目其实拿到了正确的中文标题。另外三篇文献则显示
`deterministic · translation_unavailable · 内容解读：failed`，即翻译在三个
provider 上全部失败，落回未翻译状态。

**根因**：`pipeline.py::_translate_and_analyse` 原来先跑“深度解读”
（`_run_analysis_for_work/_run_analysis_for_article`），解读任务里绑定了翻译，
且解读的语义校验（不支持的论断、证据 ID 等）比纯翻译校验严格得多。当解读校验
失败但绑定的翻译本身合格时，旧代码会把这次“翻译副产品”写回
`item["translation_audit"]`，随后解读还会继续尝试下一个 provider，新的失败
记录会追加进同一条 `attempt_chain`，于是审计栏里出现“先成功后失败”的自相矛盾
记录。更严重的是：如果解读任务在三个 provider 上全部连翻译一起校验失败，条目
就会完全没有翻译——即使一次干净的纯翻译调用本来很可能成功。

**修复**：
- 纯翻译阶段 `_translate_remaining_items` 提前到深度解读之前执行。
- 深度解读阶段不再调用 `apply_translation()`，不再有任何机会覆盖或清空已经
  生效的 `translation_audit`；它现在只负责写入 `ai_analysis` 与
  `processing_audit.llm_analysis`。
- 移除了解读阶段兜底覆盖翻译的分支，翻译状态从此只有一个权威来源。

（`src/pdi/pipeline.py`）

## 3. 翻译校验：数字千分位分组误判

**现象**：一篇讨论汉坦病毒的印度重症医学综述（原文含 “approximately 2,00,000
reported cases”，采用印度式两位数字分组）在三个 provider 上全部翻译失败。

**根因**：`translation.py::_canonical_number_token` 的千分位识别只认西式三位
分组（`len(part) == 3`），像 “2,00,000” 这种非三位分组的数字会被当作“原样保留
逗号”的字面 token，要求译文必须逐字符包含 “2,00,000”——但任何正常翻译都会
把它写成 “200,000” 或 “200000”，于是每次都会触发
`TITLE_NUMBER_CHANGED_OR_DROPPED` / `TEXT_NUMBER_CHANGED_OR_DROPPED`，
三个 provider 依次被拒。

**修复**：数字比对改为按“去除全部分组逗号后的纯数字”比较，不再要求分组方式
一致，同时仍能识别真正的数值改动（如 50 被错译成 500）。（`src/pdi/translation.py`）

## 4. 文献/新闻去重：同一事件被拆成多张卡片

**现象**：4 篇分别来自 MSN 的报道（游客须知 / WHO 宣布结束 / CDC 协调 /
乘客隔离）明明在讲同一艘游轮的同一起汉坦病毒事件，却各自生成了独立的
“公共卫生事件”卡片，页面上重复了 4 次几乎相同的信息。

**根因**：这不是文章级去重（`dedup.py::deduplicate_news`）的问题——4 篇文章
标题、角度确实不同，属于应当保留的独立文章；问题出在下游的事件聚合
（`events.py::cluster_events`）。聚合阈值需要综合相似度 ≥ 0.80，但：
- 两篇文章都没有解析出国家实体时，相似度函数只给 0.45（“地点待核验”是本次
  几乎所有事件的常态），与“两个国家明确不同”的惩罚力度几乎一样重；
- `event_type` 分类只要不在同一“家族”（如 outbreak vs other），相似度直接
  硬归零，哪怕病原体、标题、日期高度一致。
两者叠加，把综合分数持续压在阈值以下。

**修复**：
- 双方都未解析出国家时，相似度从 0.45 提高到 0.6（仍明显低于“国家一致”的
  1.0，避免过度合并真正不相关的事件）。
- `event_type` 家族不一致不再硬归零，改为 0.1 的惩罚分，允许病原体、标题、
  日期证据把真正同源的报道拉过合并阈值。

（`src/pdi/events.py`）

**已知局限**：本次修复用简化脚本验证，两篇同为 `outbreak` 标签的文章相似度
已经明显提升、非常接近阈值；`other` 标签的文章因标题用词差异较大，可能仍需
你观察下一次真实运行的 `data/audit/object_audit.jsonl` 来判断是否需要再把
阈值从 0.80 略降（如 0.78），不建议在没有真实数据反馈前继续盲调。

## 5. 文献解析 / 内容抓取：MSN 等聚合站点正文抓取必然失败

**现象**：4 篇 MSN 文章全部显示 “Original abstract or excerpt is unavailable”，
说明多层正文抓取兜底（JSON-LD → article/main → 段落 → AMP/canonical 链接
→ RSS 摘要）在 MSN 上全部落空。

**根因**：MSN 对未渲染 JS 的普通 HTTP 客户端返回的是内容为空的 JS 壳页面，
不是能被解析的静态 HTML；而且页面里的 canonical/AMP 链接通常还是指回 MSN
自己，导致三次跳转重试全部浪费在同一个死胡同上，还耽误了预算。

**修复（有限度）**：给 `enrichment.py::_fetch_one_news` 增加了一个已知
“JS-only” 域名清单（目前只有 `msn.com`），命中时直接跳过三次爬取重试，
第一时间尝试 RSS 摘要兜底，并把失败原因明确标注为
`KNOWN_JS_RENDERED_DOMAIN_NO_HEADLESS_BROWSER`，而不是含糊的
“NO_USABLE_PAGE_OR_SNIPPET”。

**未修复、需要你决策**：要真正抓到 MSN 类站点的正文，需要引入无头浏览器
（如 Playwright），这会明显增加依赖体积和 GitHub Actions 运行时间/资源
消耗。这是一个成本明显的架构变更，本轮没有替你做这个决定，先诚实标注
限制，等你确认是否值得为少数几个域名引入无头浏览器依赖。

## 回归测试说明

本环境无法访问外网安装 `rapidfuzz`、`streamlit` 等依赖，因此这一轮没有条件
跑通仓库自带的 `pytest` 套件。已完成的验证：

- 全部改动文件通过 `python -m py_compile` 语法检查。
- `_canonical_number_token` 用一组含印度分组、西式分组、无分组、小数、
  百分比的用例做了单元级人工验证，行为符合预期。
- `events._similar` 用简化的 token 相似度桩函数验证了修复方向正确（同类型
  文章相似度明显提升）。

**强烈建议在合并前于本地或 CI 跑一次**：

```bash
python -m pip install -r requirements.txt
pytest tests/test_translation.py tests/test_events.py tests/test_dedup.py \
       tests/test_pipeline.py tests/test_v16_translation_quality_local_mt.py -v
python scripts/validate_project.py
python scripts/run_daily.py --profile hantavirus --output-dir build/demo --demo --disable-llm
```
