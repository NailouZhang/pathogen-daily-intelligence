# 实现与审计注意事项

## 已实现的 v1.1 边界

本版实现了从中英查询计划到 GitHub 数据分支持久化和 Streamlit 展示的完整链路，并提供 Demo、CI、状态恢复、失败降级、静态 HTML 下载和多页面可视化。

它不把人工种子伪装成 ICTV 最终清单，也不承诺所有国家卫生机构已经完成注册。

## 关键事实边界

- 正式分类、物种状态和改名关系只能由 ICTV 等权威来源确认；
- DOI、PMID、卷期、病例数、日期、地点和宿主不得由模型猜测；
- 未知数字为 `null`，只有来源明确报告零时才能保存 `0`；
- 新闻发现器用于发现线索，不自动成为 A 级事实来源；
- 无摘要文献只显示元数据，不根据标题生成长摘要；
- Demo、缓存和生产数据必须在界面中明确区分。

## 事件聚类

当前聚类使用病原、国家、事件类型、日期和标题相似度，并以较高阈值自动合并。方案有意偏向 precision。

必须人工复核：

- 同城不同日期的独立病例；
- 人类病例与啮齿动物监测；
- 同省不同县事件；
- 媒体背景文章与当日通报；
- 相同病例数字但时间不同；
- 官方否认或数字更正。

## LLM 安全

网页正文属于不可信数据。模型提示词禁止把正文中的命令当成系统指令。模型结果经过 JSON、evidence ID、supporting item ID 和数字检查。验证失败时丢弃 AI 结果并保留原始摘要或规则结果。

当前数字校验较保守；复杂百分数、范围和格式化数字可能产生误拒绝。系统优先误拒绝而非误发布。

## API 与来源

- PubMed、Europe PMC、Crossref 和 Semantic Scholar 均为独立发现源；
- Semantic Scholar 额度、GDELT 稳定性、Google News RSS 中转和 WHO 页面结构可能变化；
- `source_health` 区分成功有结果、成功无结果、部分完成、失败和未启用；
- CDC/ECDC 仍是未启用候选，待测试稳定的病种页面、RSS、API 或 sitemap 后再启用。

## Git 与 Streamlit 存储

`intelligence-data` 适合中小规模最终实体和日报历史。原始响应应优先作为短期 Actions artifact，而不是无限写入 Git。

Streamlit 的 `runtime/dashboard_cache` 只是临时读取缓存：

- 不提交 Git；
- 容器重启可能消失；
- 不能作为正式数据库；
- UI 必须标明当前是否使用缓存。

## 生产前检查

1. 设置真实联系邮箱和 User-Agent；
2. 审查 Profile、词典和来源注册表；
3. 以 Demo 模式验证数据分支、artifact 和 Streamlit；
4. 确认 Workflow 不再包含 GitHub Pages action；
5. 确认 Streamlit 首页显示 GitHub 生产数据；
6. 以真实模式连续运行至少 14–28 天；
7. 人工抽查漏检、误合并、病例数字和来源等级；
8. 构建事件聚类金标准集；
9. 记录每次 Schema、规则、提示词和 Profile 版本变化；
10. 监控仓库体积、API 消耗和 token 过期时间。
