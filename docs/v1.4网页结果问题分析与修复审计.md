# v1.4 网页结果问题分析与修复审计

## 1. 对 2026-07-15 已发布网页的检查

对用户保存的 GitHub Pages 网页进行结构化检查，共发现 56 个卡片：15 篇文献、41 个公共卫生事件。18 条记录在 GitHub Models、Gemini、Groq 全部尝试后仍无通过校验的中文翻译；47 个卡片没有通过证据校验的深度分析。

页面中还存在多类明显的编辑污染：

- Ebola 报道仅在正文末尾回顾一次汉坦病毒游轮事件，却被生成汉坦病毒事件，并错误继承 Ebola 病例数和国家。
- BBC 国家韧性、纽约灭鼠、讣告、综合健康通讯等文章只是在背景中提到汉坦病毒，却进入“疫情与公共卫生”。
- 部分页面只获得标题或 RSS 摘要，但卡片没有始终明确区分“正文已抓获”和“正文未抓获”。
- 部分文献存在当前上线日期与未来期刊卷期日期混用；未来卷期可能导致窗口判断偏差。
- 至少一条文献出现题名与 DOI 显著不匹配，说明旧版跨来源合并过度信任共享标识符。

## 2. 根因

### 2.1 新闻背景提及被当成主体

旧版在事件聚类之前，只检查目标病原是否在全文任意位置出现。一次背景提及也会进入聚类。实体提取又在较宽上下文中寻找国家和病例数字，因此其他疾病的数字可能污染目标病原事件。

### 2.2 正文提取策略与覆盖状态不充分

不同网站正文位于 JSON-LD、`article`、`main`、自定义正文容器、AMP 页面或元描述。旧版虽然已有多种提取方式，但事件过滤发生得较晚，且未对“标题/RSS 摘要”和“正文证据”建立足够严格的分析边界。

### 2.3 文献日期使用出版排期而非当前可获得时间

期刊可提前分配未来卷期。仅按 print/issue 或通用 publication date 检索，会遗漏已经在线发表、刚进入数据库或刚创建 Crossref 记录的文献，也可能把未来排期误当成当前发表日期。

### 2.4 跨来源标识符合并过强

同一 DOI/PMID 被某个来源错误映射时，旧逻辑可能把标题完全不同的记录合并。错误 DOI 随后会覆盖正确书目信息。

### 2.5 模型兜底只在提供商层面，不充分覆盖“有效 JSON 但校验失败”

Groq 模型列表可以动态获取，但旧路径主要在 API/JSON 请求失败时轮换模型。一个模型返回合法 JSON、但翻译丢数字或证据校验失败后，可能直接结束该提供商，而没有尝试 Groq 的下一个活跃模型。

## 3. v1.4 修复

### 3.1 当前可报道日期

文献保留所有原始日期，但使用以下优先级决定当期归属：

1. online/electronic；
2. first publication；
3. PubMed/Crossref created、indexed、entry、deposited；
4. publication；
5. print/issue。

未来 print/issue 日期只在页面显示为“期刊卷期日期”，不用于判定当期新增。

### 3.2 文献多源增量发现

- PubMed：Publication Date、EPDAT、CRDT、EDAT 联合检索。
- Europe PMC：FIRST_PDATE 与 E_PDATE。
- Crossref：created-date 与 online-pub-date 双路径。
- Semantic Scholar：日期窗口后再执行本地相关性校验。

### 3.3 多策略正文获取与显式空值

新闻依次尝试：

- JSON-LD `articleBody`；
- `article`、`main`、`role=main`；
- 常见正文 CSS 容器；
- 段落回退；
- canonical/AMP；
- 元描述；
- RSS/source snippet。

页面明确显示：

- 已获得相关正文；
- 仅获得部分相关正文；
- 仅标题/RSS 摘要；
- 未抓获正文。

没有正文时，不执行正文级大模型分析。

### 3.4 先过滤再聚类

新闻先根据标题、首段、全文提及次数和上下文提示词进行相关性分类。仅 `headline/brief/review` 且病原为主体的文章进入事件聚类。背景提及直接归档。

### 3.5 病例数与地点上下文约束

病例数、死亡数、国家和事件类型只从明确包含目标病原或疾病名称的证据句中抽取。Ebola、黄热病或其他疾病的数字不会再自动附加到汉坦病毒事件。

### 3.6 保守文献合并

共享 DOI/PMID 仍需通过标题、作者和日期相容性检查。冲突记录保持独立并进入复核，不允许错误标识符静默合并。

### 3.7 动态多模型轮换

每个提供商在运行时发现可用模型。Groq 会排除语音、守卫、嵌入和审核模型，并按翻译或分析任务排序。模型出现以下情况时，尝试同一提供商的下一个模型：

- API 或模型不可用；
- 无效 JSON；
- 输出缺少记录；
- 翻译数字、科学标签或字段校验失败；
- 证据 ID 或事实校验失败。

只有该提供商的模型尝试预算耗尽后，才进入下一提供商。

### 3.8 翻译校验改善

千位分隔符可按中文习惯变化，例如 `1,600` 与 `1600` 视为同一数值；但数字本身、百分号、小数、比较符和单位不能改变。

## 4. 审计边界

- 不绕过登录、付费墙、验证码或访问限制。
- 不保存完整受版权保护网页，仅保存有限相关证据片段和原始链接。
- 没有摘要或正文时不根据标题编造内容。
- 模型失败不会阻止 JSON、网页、RSS、邮件和历史归档发布。
- 所有模型、模型轮换、校验错误和正文抓取状态写入审计文件。
